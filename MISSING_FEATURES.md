# Recommended Missing Features

After a thorough analysis of typack's codebase and comparison with competing tools (`@microsoft/api-extractor`, `dts-bundle-generator`, `rollup-plugin-dts`, `tsup`), here are features we should consider supporting, grouped by priority.

---

## High Priority

### 1. Glob/Pattern-based External Matching

**Current:** `external` only supports exact string match (e.g., `"react"`).
**Proposed:** Support glob patterns like `@scope/*`, `react-*`, or regex patterns.

`dts-bundle-generator` supports `--external-types` with pattern matching for scoped packages. This is essential for monorepos and scoped packages where listing every sub-package is impractical.

```rust
// Example: external: ["@myorg/*", "react", "react-*"]
```

### 2. Config File Support

**Current:** All options must be passed via CLI flags or the programmatic API.
**Proposed:** Support a `typack.config.json` / `typack.config.ts` config file, or reading config from `package.json`'s `"typack"` field.

Every serious bundler supports config files. This is critical for reproducible builds and CI/CD pipelines where complex option sets are common.

### 3. Multiple Output Files (One Per Entry)

**Current:** Multiple entry points are all bundled into a single output file.
**Proposed:** Support generating one output file per entry point, mapping `input[i]` → `output[i]`.

Libraries like Vue have multiple entry points (`vue`, `vue/reactivity`, `vue/runtime-core`) that should produce separate `.d.ts` bundles. This is a common pattern in modern library authoring.

### 4. tsconfig.json Path Alias Resolution

**Current:** Resolution uses `oxc_resolver` with default options + `"types"` condition.
**Proposed:** Read `tsconfig.json` to respect `paths`, `baseUrl`, and `rootDir` mappings.

Many projects use path aliases (`@/components/*` → `src/components/*`). Without tsconfig support, these imports fail to resolve.

---

## Medium Priority

### 5. `@internal` / Release Tag Trimming

**Current:** No support for filtering declarations by JSDoc tags.
**Proposed:** Support stripping declarations tagged with `@internal`, `@alpha`, or `@beta` from the output.

This is `api-extractor`'s most unique feature. It's valuable for libraries that want to expose a public API surface while keeping internal types hidden from consumers. A simpler version (just `@internal` stripping) would cover the most common use case.

### 6. Banner / Footer Support

**Current:** No way to add custom content to the output.
**Proposed:** Support `banner` and `footer` options to prepend/append custom text (e.g., license headers, auto-generated warnings).

```rust
TypackOptions {
    banner: Some("/* Auto-generated. Do not edit. */".into()),
    ..
}
```

### 7. Watch Mode

**Current:** CLI runs once and exits.
**Proposed:** A `--watch` flag that monitors input `.d.ts` files and re-bundles on change.

Essential for development workflows where `.d.ts` files are continuously regenerated by `tsc --watch` or similar tools. This would make typack more useful as a standalone development tool.

### 8. `--no-check` / Lenient Mode

**Current:** Unresolved relative specifiers are fatal errors.
**Proposed:** A lenient mode that downgrades resolution errors to warnings and continues bundling.

Useful for incremental adoption where not all dependencies have proper `.d.ts` files yet, or when bundling partial type definitions.

---

### 9. UMD `export as namespace` Wrapping

**Current:** No support for UMD global namespace output.
**Proposed:** A `--umd-module-name <NAME>` option to append `export as namespace <NAME>` to the output.

`dts-bundle-generator` supports `umdModuleName` for libraries that expose a global UMD namespace. This is needed for libraries consumed via `<script>` tags.

---

## Lower Priority (Nice to Have)

### 10. Output Validation

**Current:** No post-bundle validation.
**Proposed:** Optionally re-parse the output `.d.ts` to verify it's syntactically valid TypeScript.

This would catch subtle codegen bugs early. Could be a `--validate` flag or an internal debug assertion.

### 11. Inline Node Modules Types

**Current:** `node_modules` dependencies are always kept external.
**Proposed:** An `--inline` option (inverse of `--external`) to inline type definitions from specific `node_modules` packages.

`dts-bundle-generator` supports `--external-inlines` for this. It's useful when a dependency's types are small and you want a self-contained `.d.ts` with zero external dependencies.

### 12. JSON/Diagnostic Output Mode

**Current:** Warnings and errors are printed as human-readable text to stderr.
**Proposed:** A `--format json` flag to output structured diagnostics as JSON.

Useful for IDE integrations, CI pipelines, and programmatic consumers that need machine-parseable output.

### 13. Preserve/Strip Comment Options

**Current:** Comments are preserved as-is from the input.
**Proposed:** Options to control comment handling: `--strip-comments` to remove all comments, or `--preserve-jsdoc` to keep only JSDoc comments while stripping others.

`dts-bundle-generator` has `--no-banner` and comment control options. This is useful for reducing output size in production bundles.
